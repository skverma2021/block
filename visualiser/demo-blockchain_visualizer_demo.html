SHA-256 converts any input into a fixed 64-character hexadecimal string called a hash. The same input always produces the same hash. There is no way to reverse the hash back to the original data.

In our system: every sensor reading is hashed on arrival. The hash (rowHash) is stored alongside the data. If the reading is altered — even by one digit — the hash will not match and the tamper is detected immediately.
The avalanche effect means a tiny change in input causes roughly half the output digits to change. You cannot nudge the hash with a small change — any modification produces a completely unpredictable new hash.

Each block includes the hash of the previous block as part of its own content. This links blocks into a chain — you cannot silently change an old block without also changing every block that follows it.

In our system: blockHash = SHA-256(index + timestamp + merkleRoot + prevHash + nonce + txIds). Changing any field — including prevHash — produces a completely different hash.
In a real blockchain the previous hash stored in each block is frozen at the moment the network accepted it. If someone changes an old block's data later, that block's hash changes — but the next block still holds the old (now wrong) hash. The mismatch is visible to anyone who checks.

In our system: verifyChainIntegrity() re-computes every block hash from scratch and compares it to the stored value. Any mismatch triggers a CHAIN_INTEGRITY_FAILURE alert in the audit_log table, and the system can purge and resync from the Regulatory Authority node.
A Merkle tree fingerprints all the transactions within a single block into one value — the Merkle Root. This root is stored in the block header. If any individual transaction is altered, the root changes, the block hash changes, and the chain breaks.

In our system: merkleRoot = SHA-256 tree over all rowHash values in a block. Because the Merkle Root is part of the block hash formula, altering any transaction changes the root, changes the block hash, and breaks the chain.
Proof of Work (PoW) blockchains require miners to find a nonce that makes the block hash start with a set number of zeros. This is computationally expensive but easy to verify.

Our system uses Proof of Authority (PoA) — the Regulatory Authority mines by right, not by computation. The nonce is always 0. No puzzle required.
